## Контекст
Система состоит из нескольких независимых микросервисов (`OrderService`, `ProductService`, `PriceService`), которые должны согласованно обрабатывать создание заказа.

Поскольку сервисов мало (3), бизнес-поток линейный, а команда небольшая, необходимо выбрать решение с минимальным операционным оверхедом, которое легко поддерживать и отлаживать.

## Решение
Реализовать хореографическую SAGA с использованием `MassTransit` и `Kafka`. Каждый сервис самостоятельно реагирует на интеграционные события и публикует свои.

Ключевые механизмы:
- **Идемпотентность**: Все потребители проверяют `EventId` перед обработкой, чтобы избежать дублей при ретраях Kafka
- **Таймаут 30с**: MassTransit Scheduler публикует `SagaTimeoutEvent`. Если ответы не пришли вовремя → заказ автоматически отменяется
- **Компенсация**: При отказе любого шага публикуется `OrderCancelled`, который откатывает изменения в других сервисах (освобождение резерва товаров)
- **Асинхронный API**: `POST api/orders` возвращает `202 Accepted` + `statusCheckUrl`. Финальный статус определяется через `GET /api/orders/{id}/status` 

## Последствия
### Позитивные
- Отсутствие единой точки отказа (нет Orchestrator)
- Слабая связанность: сервисы не знают о существовании друг друга, только о контрактах событий
- Простота реализации и отладки для малого количества сервисов

### Негативные
- eventual consistency: между `OrderCreated` и `OrderConfirmed` есть задержка до 30с
- Сложность трассировки: для отладки распределённого потока требуется `CorrelationId` и централизованные логи
- Необходимость писать идемпотентные обработчики и корректно обрабатывать таймауты
- Риск "подвешенных" заказов при потере событий (снижается через DLQ, ретраи и периодическую сверку)

### Влияние на Frontend
- UI должен работать в асинхронном режиме: после `202` показывать состояние `"Обрабатывается..."`
- Реализовать пулинг `/api/orders/{id}/status`
- Корректно обрабатывать конечные состояния: `Confirmed` - редирект на страницу заказа, `Failed`/`Timeout` - показать ошибку + кнопку "Повторить"

## Рассмотренные альтернативы
| Вариант | Почему отклонён |
|---|---|
| **Оркестратор (Saga Orchestrator)** | Избыточная сложность для 3 сервисов. Требует отдельного компонента/библиотеки для управления состоянием и маршрутизации событий. |
| **Синхронные HTTP-вызовы (REST)** | Приводит к каскадным отказам, блокирует потоки, не масштабируется. При таймаутах требует ручной компенсации и сложной обработки сетевых разрывов. |
| **Распределённые транзакции** | Не поддерживается в микросервисной архитектуре, создаёт жёсткую связность, блокирует ресурсы БД, не масштабируется горизонтально. |
