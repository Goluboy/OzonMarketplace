# ADR-006: Стратегия управления топиками Kafka (1 топик на бизнес-флоу)

|---|---|

| **Статус** | `⚪ Заменено (ADR-007)` |
| **Авторы** | `@Goluboy` |

## Контекст
Для реализации choreography SAGA необходимо организовать Kafka. Требуется выбрать стратегию именования и разделения топиков, которая обеспечит:
1. Простоту отладки и трассировки одного бизнес-процесса.
2. Минимальный операционный оверхед для команды из 3 человек.
3. Безопасную маршрутизацию событий (чтобы сервисы не обрабатывали "чужие" сообщения).
4. Возможность эволюции контрактов (версионирование).

## Решение
Использовать стратегию **1 топик на бизнес-флоу»*.

Все события, связанные с процессом создания заказа, публикуются в **единый логический топик** `orders`, независимо от того, какой сервис их сгенерировал.

### Карта событий для топика `orders`:

| Событие | 📤 Издатель | 📥 Подписчик(и) | Назначение в SAGA |
|---------|-------------|-----------------|-------------------|
| `OrderCreatedEvent` | `OrderService` | `ProductService`, `PriceService` | Запуск параллельных шагов (резерв + цена) |
| `StockReservedEvent` | `ProductService` | `OrderService` | Подтверждение резерва товара |
| `StockReservationFailedEvent` | `ProductService` | `OrderService` | Сигнал об ошибке резерва |
| `PriceCalculatedEvent` | `PriceService` | `OrderService` | Фиксация итоговой суммы |
| `OrderCancelledEvent` | `OrderService` | `ProductService` | Компенсация: отмена резерва |

### Техническая реализация (MassTransit):
Маршрутизация осуществляется на уровне клиента через заголовок **`MT-MessageType`**.
1. **Публикация**: Издатель пишет сообщение в топик `orders`. MassTransit автоматически добавляет заголовок с полным именем типа (например, `OrderService.IntegrationEvents.OrderCreatedEvent`).
2. **Потребление**: Потребитель подписывается на топик `orders`, но указывает конкретный тип: `k.TopicEndpoint<OrderCreatedEvent>("orders", ...)`.
3. **Фильтрация**: При получении сообщения клиент проверяет заголовок. Если тип не совпадает с ожидаемым — сообщение игнорируется.

## Последствия
### Позитивные
- **Упрощённая отладка**: Чтобы восстановить полную картину выполнения заказа, достаточно прочитать один топик `orders` и отфильтровать сообщения по `CorrelationId`. Не нужно агрегировать логи из 5 разных топиков.
- **Операционная простота**: Меньше объектов для управления (мониторинг лагов, настройка retention, бэкапы, ACL). Для команды из 3 человек это критически снижает когнитивную нагрузку.
- **Логическая связность**: Все события одного процесса физически находятся рядом, что отражает бизнес-реальность.
- **Гибкость версионирования**: Легко поддерживать сосуществование V1 и V2 событий в одном потоке (разные `MT-MessageType`).

### Негативные / Риски
- **Head-of-Line blocking (может быть)**: Если один тип события начнёт генерировать 99% трафика, он может создать очередь, которая задержит доставку редких событий.
  - *Митигация*: Мониторинг размера очереди. При возникновении проблемы — вынести "шумное" событие в отдельный топик.
  - **Единая политика хранения**: Нельзя задать разный `retention.ms` для разных типов событий внутри одного топика.
  - *Митигация*: Выбирать настройку retention, достаточную для самого "долгоживущего" сценария отладки.

### Влияние на Frontend
Отсутствует

## Рассмотренные альтернативы
| Вариант | Почему отклонён |
|---|---|
| **1 топик на тип события**<br>(`orders.created`, `orders.stock_reserved`...) | Создает избыточную сложность для отладки: трассировка одного заказа требует чтения 5+ топиков. Увеличивает операционный оверхед (настройка 50+ топиков вместо 3-5). |
| **1 топик на сервис**<br>(`order-service`, `product-service`...) | Нарушает логическую целостность флоу. События `OrderCreated` (из OrderService) и `StockReserved` (из ProductService) оказываются в разных топиках, что усложняет анализ причинно-следственных связей. |
| **Schema Registry (Avro)** | Избыточно для текущей скорости разработки. JSON + `MT-MessageType` обеспечивает достаточную типобезопасность и гибкость для команды из 3 человек. |
