# ADR-006: Стратегия управления топиками Kafka (1 топик на бизнес-флоу)

| Метаданные | Значение |
|---|---|
| **Дата** | `2026-06-02` |
| **Статус** | `🟢 Принято` |
| **Авторы** | `@Goluboy` |
| **Затронутые компоненты** | `OrderService, ProductService, PriceService, Kafka, MassTransit` |

---

## Контекст
MassTransit запрещает регестрировать несколько продюсеров на 1 топик, т.к. использует название топика как ключ под капотом.
возможные опции:
 - использовать 1 топик на 1 класс
 - Avro + Schema Registry (по звявлению автора masstransit - это единственный способ регестрировать несколько ивентов на топик)
 - Отказаться от MassYransit.

## Решение
Использовать 1 топик на 1 класс. и оставить MassTransit.

### Карта событий:

| Событие                       | Издатель         | Подписчик(и)                     | Топик Kafka               | Назначение в SAGA                         |
| ----------------------------- | ---------------- | -------------------------------- | ------------------------- | ----------------------------------------- |
| `OrderCreatedEvent`           | `OrderService`   | `ProductService`, `PriceService` | `orders.created`          | Запуск параллельных шагов (резерв + цена) |
| `StockReservedEvent`          | `ProductService` | `OrderService`                   | `products.stock.reserved` | Подтверждение резерва товара              |
| `StockReservationFailedEvent` | `ProductService` | `OrderService`                   | `products.stock.failed`   | Сигнал об ошибке резерва                  |
| `PriceCalculatedEvent`        | `PriceService`   | `OrderService`                   | `prices.calculated`       | Фиксация итоговой суммы                   |
| `OrderCancelledEvent`         | `OrderService`   | `ProductService`                 | `orders.cancelled`        | Компенсация: отмена резерва               |

### Техническая реализация (MassTransit):

Маршрутизация осуществляется на уровне клиента через заголовок **`MT-MessageType`**.
1. **Публикация**: Издатель пишет сообщение в свой топик.
2. **Потребление**: Каждый консюмер привязан к конкретному топику через TopicEndpoint<T>. MassTransit автоматически десериализует сообщения в указанный тип. Фильтрация по типу сообщения не требуется — Kafka сам распределяет сообщения по топикам.("orders", ...)`.

### Соглашения по именованию топиков

Используется иерархическая структура с точкой как разделителем:
- `{домен}.{действие}` для событий домена: `orders.created`, `orders.cancelled`
- `{домен}.{сущность}.{действие}` для более детальных событий: `products.stock.reserved`
- Суффиксы версионирования при необходимости: `orders.created.v2`

## Последствия
### Позитивные
- Совместимость с MassTransit: Отсутствие конфликтов при регистрации продюсеров, работа "из коробки" без кастомных сериализаторов.
- Изоляция потоков: Медленный потребитель одного типа события не блокирует обработку других типов (отсутствие head-of-line blocking между разными событиями).

### Негативные / Риски

- Увеличение количества топиков: Вместо 1-3 топиков на домен получается 5+ топиков на один бизнес-флоу.
 - Митигация: Для 3 сервисов и 5 типов событий общее количество топиков (10-15) остаётся в пределах управляемого диапазона. При росте до 50+ топиков потребуется автоматизация
- Усложнение трассировки одного заказа: Для восстановления полного флоу создания заказа необходимо читать из 3-5 топиков и агрегировать сообщения по CorrelationId.
 - Митигация: Внедрение распределённой трассировки (OpenTelemetry + Jaeger), где все события связываются по trace_id. Kafka UI поддерживает фильтрацию по заголовкам сообщений.
- Нагрузка на брокер: Каждый топик создаёт оверхед. Но для текущих нагрузок это не проблема

### Влияние на Frontend
Отсутствует

## Рассмотренные альтернативы
| Вариант                                                     | Почему отклонён                                                                                                                                                                                                                                                                                                                                                                          |
| ----------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1 топик на домен (все события в `orders`)**               | MassTransit Kafka Rider использует имя топика как ключ регистрации `ITopicProducer<T>`. Регистрация нескольких продюсеров (`OrderCreatedEvent`, `OrderCancelledEvent`) для одного топика вызывает `ConfigurationException: A topic producer with the same key was already added`. Обход через динамические продюсеры (`ITopicProducerProvider`) усложняет код и теряет типобезопасность. |
| **Avro + Schema Registry (несколько типов в одном топике)** | Требует внедрения Schema Registry как отдельного сервиса, перехода на бинарный формат Avro, написания кастомных сериализатор/десериализатор. Автор MassTransit описывает это как "hacky" решение с ограниченной поддержкой. Для команды из 3 человек и 5 типов событий операционный оверхед не окупается.                                                                                |
| **Отказ от MassTransit, прямая работа с Confluent.Kafka**   | Лишает команду встроенных механизмов retry, checkpoint, DI-интеграции для консюмеров. Требует реализации собственного слоя абстракции для идемпотентности, обработки ошибок и graceful shutdown.                                      |
| **RabbitMQ вместо Kafka**                                   | RabbitMQ поддерживает множественные типы сообщений в одном exchange через routing keys. Однако переход на другой брокер требует переписывания инфраструктуры, потери опыта работы с Kafka, и не соответствует долгосрочной стратегии использования event streaming для аналитики и event sourcing.                                                                                       |
