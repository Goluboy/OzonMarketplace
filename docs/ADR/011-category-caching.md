## Контекст
Поскольку категории должны иметь возможность изменения, было принято решение хранить их в базе.
Категории меняются довольно редко, за счёт этого нужно было разработать оптимальную стратегию для их кеширования на клиенте.

## Решение
Кеширование на стороне браузера с меткой E-Tag, позволяющей каждый раз проверять у сервера актуальность списка категорий. Для этого на сервере
реализован механизм трёх-фазного кеширования: клиент на любой запрос отправляет также GET `/api/categories` с заголовком IF-None-Match (непосредственно сам E-Tag), 
сервер в свою очередь сверяет значение полученной версии списка категорий с актуальной версией в Redis. Если значения совпадают - возвращается 304, данные на клиенте актуальны.
Если ключи разные (кеш на клиенте не актуальный) - сервер проверяет актуальный список категорий в Redis и отдаёт его в случае Cache Hit. В случае Cache Miss сервер получает актуальные список
категорий из БД, генерирует новую E-Tag метку и кеширует его вместе со списком, затем отдаёт клиенту.

## Последствия
### Позитивные
- Категории на клиенте всегда актуальные
- Оптимизация работы с БД

### Негативные
- Сложность реализации
- Возможная избыточность реализации

## Рассмотренные альтернативы
| Вариант | Почему отклонён |
|---|---|
| `Без кеширования` | `Высокая нагрузка на бд` |
| `Кеширование на основе TTL` | `Вероятность неактульности данных до истечения TTL` |
