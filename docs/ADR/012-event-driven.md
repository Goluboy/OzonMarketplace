## Контекст
Для соблюдения чистоты архитектуры было принято решение добавить кеширование в сервис с помощью декораторов для репозиториев. Из-за использования паттерна UnitOfWork
инвалидировать кеш в декораторе недопустимо, поскольку работа с репозиторием идёт до закрытия транзакции. В таком случае инвалидацию кэша нужно вынести в другое место.

## Решение
Event Driven подход - решение, основанное на парадигме событий. Инвалидация кэша происходит ассинхронно с помощью библиотеки CAP, а именно во время отправки событий через outbox паттерн.

## Последствия
### Позитивные
- Кэш инвалидируется строго после коммита транзакции
- Соблюдается чистота архитектуры

### Негативные
- Согласованность к конечном счёте
- Больше накладных расходов, сильная связность от Kafka

## Рассмотренные альтернативы
| Вариант | Почему отклонён |
|---|---|
| `Отложенное удаление через UnitOfWork` | `Данный подход соблюдает строгую согласованность, но дольше в реализации. ` |
